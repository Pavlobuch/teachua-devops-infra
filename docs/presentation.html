01Project Goal

A portfolio built to show real ownership, not a checklist

  • Demonstrate end-to-end DevOps skills for interviews: take a real full-stack app and own it from local dev through to a monitored, cloud-deployed environment.
  • Deliberately sized as one cost-optimized pipeline rather than an enterprise setup — the goal is depth across the toolchain, not scale nobody needs.
  • Every infra decision — K3s vs EKS, Traefik vs ALB, Splunk's deployment shape — was made and justified explicitly. See Cost Optimization Decisions.

03Target Architecture

Two EC2 hosts, split by responsibility

App Server (K3s + workloads) and Monitoring Server (Splunk) are separate instances — isolating indexing load from production traffic and giving each an independent lifecycle.

Internet→ EC2 public IP→ Traefik→ Ingress→ Service→ Pod
App pods→ Fluent Bit→ Splunk HEC→ Dashboards / Alerts
  • Images: Jenkins builds → pushes to private ECR → K3s pulls via an imagePullSecret refreshed on every deploy.
  • Secrets: Ansible Vault at provisioning time, Jenkins credentials at pipeline time, K8s Secrets created imperatively — never committed.
04Local Dev → Docker → Compose

Local parity is what caught the first real bug

  • Multi-stage Dockerfiles: Maven build stage → slim JRE runtime for the backend; Node build → Nginx serve for the frontend.
  • Docker Compose wires frontend, backend, and MySQL together for local iteration before anything touches AWS.
  • That local-first discipline is exactly what exposed a subtle bug later: REACT_APP_ROOT_SERVER=http://localhost:8080 baked in at build time — fine on a laptop, broken the moment the same image ran against a real backend on EC2.
05AWS with Terraform

VPC, EC2, ECR, IAM — provisioned, reviewed, broken, fixed

Built

  • VPC, subnets, security groups, EC2 (App + Monitoring), ECR repos, IAM roles / instance profiles, gp3 volumes.
  • EC2 gets a role + instance profile, never embedded credentials; ECR access scoped read-only on the instance since Jenkins does the pushing.

Caught & fixed

  • A Grafana security-group rule had to_port = 22 instead of 3000 — caught in review before apply.
  • Real AWS API rejection: ECR lifecycle policy needed tagStatus = "any", not "tagged" without a filter.
  • pathexpand() needed for ~ in SSH key paths; key pair renames must happen before first apply — key_name is immutable.
06Server Provisioning with Ansible

The failures that don't throw errors are the dangerous ones

  • Five roles — common, docker, k3s_prerequisites, k3s, splunk — driven by one site.yml against a Terraform-sourced inventory.
  • Recurring pattern: silent misconfiguration, not errors. Settings outside [defaults] in ansible.cfg were silently ignored; become: true nested inside module params passed lint but did nothing; an inventory group mismatch (ec2 vs app) would have targeted zero hosts with no error at all.
  • One genuine live failure: k3s_binary.stats.exists — a typo for .stat.exists — threw a real FAILED! on the actual EC2.
  • Secrets went through Ansible Vault from the start, after an initial role plan was revised specifically because it had no vault strategy and no idempotency check.
07CI/CD with Jenkins + ECR

CI builds and pushes. CD deploys. Never the same pipeline.

  • CI and CD deliberately split into separate pipelines and repos once it was clear deploy logic didn't belong inside the backend app repo.
  • CD automates the full imagePullSecret lifecycle every run: fetch ECR token → create/update the K8s docker-registry secret → patch the default ServiceAccount → apply manifests → verify rollout.
  • Hardest bugs lived in shell-quoting across local → SSH → remote-shell boundaries: an ECR password quoting bug I first got wrong and reversed after re-tracing it, and a kubectl patch JSON payload that arrived stripped of its quotes.
  • Building on Apple Silicon and deploying to an x86_64 EC2 node caused ImagePullBackOff — fixed with an explicit --platform linux/amd64.
08Kubernetes on K3s

Every YAML file had a case-sensitivity trap the first time through

  • Namespace, Deployments, Services, ConfigMaps, Secrets, Ingress — hand-authored, applied via kubectl apply -R.
  • stringdata vs stringData, targetport vs targetPort, mistyped matchLabels — most caught in review; one metadata:: double-colon actually threw resource name may not be empty on the live cluster.
  • K3s already ships Traefik on 80/443, so an AWS ALB was deliberately skipped — see Cost Optimization.
  • Acknowledged tech debt: MySQL/JWT secrets are still plaintext K8s Secrets — External Secrets Operator, Sealed Secrets, and Vault Agent Injector were all scoped, none implemented yet.
09Monitoring with Splunk

Replaced Prometheus + Grafana — on purpose, not by accident

  • One tool covering logs and dashboards is simpler to demo end-to-end in an interview than a two-tool stack — and Splunk/SPL is more directly relevant to enterprise ops/security roles.
  • Runs on its own dedicated EC2, not inside K3s — indexing is memory/CPU/disk-hungry and would contend with the app and database on one box.
  • Fluent Bit ships container logs over HEC; a Universal Forwarder ships host metrics from four self-authored scripts, after Splunk's official TA-nix add-on turned out to be gated behind a Splunkbase login.
  • A manifest apply-order bug (Fluent Bit files sorting before namespace.yaml) was fixed by prefixing the namespace file 00-; a near-miss *.example.yaml secret rename — which kubectl apply -R would still have applied — was caught before it could leak a placeholder token into the cluster.
10Problems Solved / Troubleshooting

Real incidents, root-caused from actual error output

01 — ansible

Inventory swap put Splunk and K3s on the wrong boxes. Caught via a Jenkins kubectl: command not found failure; fixed by correcting inventory and recreating the contaminated EC2 via Terraform.

02 — terraform

ECR lifecycle policy rejected by AWS. tagStatus=TAGGED needs a tag filter; fixed with tagStatus=any.

03 — terraform

Security-group rule caught pre-apply. Grafana's to_port was 22 instead of 3000 — would have blocked monitoring while duplicating SSH exposure.

04 — jenkins

Cross-architecture image builds. arm64 images built on Apple Silicon failed on the amd64 node; fixed with --platform linux/amd64.

05 — jenkins

Shell-quoting lost across the SSH boundary. Both an ECR password and a kubectl patch JSON payload lost quoting crossing local → SSH → remote shell.

06 — k3s

Apply-ordering bug. Fluent Bit manifests applied before the monitoring namespace existed; fixed by forcing the namespace file first.

07 — k3s

Near-miss secrets leak. A *.example.yaml rename would still have been applied by kubectl, writing a placeholder token in as a real credential.

08 — splunk

Package collision after the inventory swap. Re-running Ansible hit an init.d boot-start collision; resolved with a clean Terraform recreate.

09 — splunk

Private IP instability across recreates. Fluent Bit's Splunk target IP is a literal value in a committed ConfigMap — no static pinning meant every recreate silently broke log shipping until fixed.

11Cost Optimization Decisions

Every tradeoff was a decision, not a default

K3s > EKS

A managed control plane is unnecessary spend for a project sized to demonstrate skills, not serve real traffic.

Traefik > ALB

Saves roughly $20/month — K3s already binds 80/443 on the host, so a managed load balancer would be redundant for one node.

Splunk, one EC2

Chosen for demo simplicity and portfolio relevance over a Prometheus + Grafana pair — still isolated from the app box to avoid resource contention.

ECR lifecycle

Simplified to two rules — expire untagged after 14 days, keep the last 10 tagged — sized for portfolio churn, not copied from a production tier.

t3.medium, kept

Flagged mid-project as oversized for the workload; kept for stability during active development — a deliberate "not yet" call, not an oversight.

SSM, reconsidered

Initially rejected as unnecessary complexity on top of SSH + Ansible, then added back after reconsideration — a real design conversation, not default-everything.
